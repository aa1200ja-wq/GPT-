# Codex Cloud 專案執行規則範本

> 使用本範本前，必須先替換所有 `<...>` 佔位文字，不得直接原樣使用。

## 一、專案基本設定

- 唯一允許操作的儲存庫：`<OWNER>/<REPOSITORY>`
- 禁止操作的儲存庫：`<PROTECTED_REPOSITORY_OR_NONE>`
- 預設基準分支：`<BASE_BRANCH>`
- Codex Cloud 任務以介面選定的基準分支為來源
- 不得直接推送或合併至基準分支
- Pull Request 必須經 GPT／小安驗收及老闆通過後，才能合併

## 二、開始任務前

Codex 開始工作前必須：

1. 完整閱讀根目錄 `AGENTS.md`
2. 完整閱讀 `handoff/task.md`
3. 確認任務狀態明確寫為「已批准執行」
4. 確認目前只有一個允許執行的任務
5. 確認允許修改的檔案範圍
6. 確認禁止修改的檔案範圍
7. 確認驗收條件與測試要求

任務狀態不是「已批准執行」時，必須立即停止，不得修改任何檔案。

需求不清楚或資料不足時，必須立即停止並回報，不得自行猜測。

## 三、執行限制

- 一次只處理一個原子任務
- 不得擴充任務範圍
- 不得順便執行下一個任務
- 不得修改任務未授權的檔案
- 不得重構無關程式
- 不得任意新增套件
- 不得任意刪除檔案
- 不得任意移動或重新命名檔案
- 不得破壞既有功能
- 不得寫入密碼、API Key、Token 或其他機密資料
- 修改前必須先閱讀相關檔案
- 修改後必須執行任務指定的測試
- 無法執行測試時，必須明確說明原因
- 發現可能影響其他儲存庫時，必須立即停止
- 禁止操作任務未授權的其他儲存庫

## 四、Codex Cloud 工作環境規則

- 直接使用 Codex Cloud 平台提供的工作分支
- 不得因本地沒有基準分支而停止
- 不得要求存在 `origin` remote
- 不得自行執行 `git fetch`
- 不得自行建立、切換或重新命名分支
- 不得自行手動推送遠端分支
- 不得直接推送或合併至基準分支
- 完成修改後保留平台產生的變更
- Pull Request 由 Codex Cloud 介面建立
- 不得自行合併 Pull Request

## 五、完成任務後

完成目前任務後，必須更新 `handoff/report.md`，至少填寫：

- 任務編號
- 任務名稱
- 執行狀態
- 執行結果摘要
- 完成內容
- 修改檔案
- 新增檔案
- 刪除檔案
- 移動或重新命名檔案
- 測試指令
- 測試結果
- 未完成項目
- 問題與風險
- 需要 GPT／小安／老闆確認的事項
- Codex Cloud 工作分支資訊
- Pull Request 資訊

若平台尚未建立 Pull Request，填寫：

`待 Codex Cloud 介面建立`

## 六、完成後停止規則

完成回報後必須立即停止。

不得：

- 自行修改 `handoff/task.md`
- 自行更改任務批准狀態
- 自行建立下一個任務
- 自行執行下一個任務
- 自行合併 Pull Request
- 自行修改基準分支
- 自行操作其他儲存庫

未經 GPT／小安驗收及老闆說「通過」，不得執行下一個任務。
